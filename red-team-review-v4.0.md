## 安全問題按優先級排序

### P1 - 嚴重問題

### P2 - 高危問題

*本版本未發現 P2 級別高危安全問題。所有 v3.4 中的 P2 問題均已修復。*

### P3 - 中危問題

#### 問題 1: `update_knowledge` 動態字段名存在 SQL 注入潛在風險
- **嚴重級別**: P3（中危）
- **涉及文件**: `vault/db.py` 第 770-778 行
- **影響功能**: 知識更新接口
- **問題描述**:
  - `update_knowledge` 方法使用 `f"{k}=?"` 動態構造 SET 子句
  - 字段名 `k` 直接來自調用方的 `**fields` 參數
  - 雖然當前所有調用鏈中的字段名都是硬編碼的，不存在直接風險
  - 但如果未來新增 MCP/API 接口允許用戶指定更新字段，可能引入 SQL 注入
  - 類似問題也存在於 `update_memory_candidate` 方法中
- **驗證代碼**:
  ```python
  # 當前調用方式（安全）：
  db.update_knowledge(kid, title="新標題", content_raw="新內容")
  
  # 潛在風險場景（如果開放用戶可控字段名）：
  # fields = {"title = 'test'; DROP TABLE knowledge; --": "value"}
  # 會導致: UPDATE knowledge SET title = 'test'; DROP TABLE knowledge; -- = ? WHERE id=?
  ```
- **修復建議**:
  1. 添加允許更新的字段白名單
  2. 對字段名進行格式驗證（只允許字母數字下劃線）
  3. 在方法文檔中明確說明字段名不可信時的風險
- **當前風險評估**: 低。所有現有調用路徑的字段名均為硬編碼，不存在可利用路徑。

#### 問題 2: 缺少 API 速率限制，存在 DoS 攻擊面
- **嚴重級別**: P3（中危）
- **涉及文件**: MCP 接口、CLI 接口
- **影響功能**: 所有公開接口
- **問題描述**:
  - 當前 MCP 和 CLI 接口沒有內置速率限制機制
  - 雖然本地優先架構降低了遠端攻擊風險
  - 但如果部署為服務或被 Agent 頻繁調用，可能導致資源耗盡
  - 特別是向量搜尋和圖譜擴展等計算密集型操作
- **影響範圍**: 性能/可用性風險
- **修復建議**:
  1. 在 MCP server 層添加速率限制（如令牌桶算法）
  2. 對計算密集型操作設置單次調用上限
  3. 添加資源使用監控和告警
- **當前風險評估**: 中低。本地優先架構下攻擊面較小。

#### 問題 3: 搜索結果默認返回完整 `content_raw`，信息洩露風險
- **嚴重級別**: P3（中危）
- **涉及文件**: 所有搜索方法
- **影響功能**: 搜索結果返回
- **問題描述**:
  - 搜索結果默認返回完整的文檔內容（`content_raw` 字段）
  - 對於大型知識庫，單條結果可能包含數千字節的敏感內容
  - 如果 MCP 接口被非預期 Agent 調用，可能導致信息過度披露
  - 雖然 MCP 本身是信任邊界內的組件，但遵循最小權限原則應優化
- **修復建議**:
  1. MCP 接口默認使用 `compact` 模式，只返回必要字段
  2. 完整內容需要顯式請求（如 `include_content=True`）
  3. 添加內容長度截斷機制
- **當前風險評估**: 中低。MCP 通常在信任邊界內使用。

#### 問題 4: 語義索引嵌入緩存缺少輸入驗證
- **嚴重級別**: P3（中危）
- **涉及文件**: `vault/semantic.py`
- **影響功能**: 語義索引、嵌入緩存
- **問題描述**:
  - `add_embedding` 等函數對輸入向量的維度驗證不嚴格
  - 雖然有 `provider_id` 和 `dimension` 參數，但主要依賴調用方正確傳入
  - 惡意構造的向量數據可能導致緩存污染或後續搜尋異常
  - 缺少對向量值範圍的基本驗證（如是否為正規化向量）
- **修復建議**:
  1. 驗證向量維度與預期一致
  2. 檢查向量值是否在合理範圍內（如 [-1, 1] 或取決於模型）
  3. 添加 NaN/Inf 檢查
- **當前風險評估**: 低。嵌入生成通常由可信模組完成。

### P4 - 低危問題

#### 問題 5: CLI 導入功能缺少路徑遍歷防護
- **嚴重級別**: P4（低危）
- **涉及文件**: `vault/cli.py`, `vault/importer.py`
- **影響功能**: 文件導入
- **問題描述**:
  - `vault import` 命令直接使用用戶提供的文件路徑
  - 沒有驗證路徑是否在預期的項目目錄內
  - 可能導致訪問系統中任意文件（如 `../../etc/passwd`）
  - 但由於是 CLI 工具，用戶本身就有文件系統訪問權限
  - 風險主要在於：如果 CLI 被包裝為 Web 服務或 API，會引入嚴重路徑遍歷漏洞
- **修復建議**:
  1. 對文件路徑進行規範化解析（`Path.resolve()`）
  2. 驗證路徑在預期的工作目錄內
  3. 添加 `--allow-absolute-path` 選項，默認關閉
- **當前風險評估**: 低。CLI 本地使用場景下風險有限。

#### 問題 6: 日誌中可能洩露敏感信息
- **嚴重級別**: P4（低危）
- **涉及文件**: `vault/log.py` 及各模組日誌調用
- **影響功能**: 日誌記錄
- **問題描述**:
  - 當前日誌系統較為簡單，沒有敏感信息過濾機制
  - 如果錯誤信息中包含 API 密鑰或敏感內容，可能被記錄到日誌
  - 特別是 LLM 調用和 Supabase 同步相關的錯誤
- **修復建議**:
  1. 添加日誌敏感信息過濾器
  2. 對常見敏感模式（API Key、Token 等）進行遮蔽
  3. 區分日誌級別，敏感信息只在 DEBUG 級別輸出
- **當前風險評估**: 低。本地部署場景下日誌訪問受控。

#### 問題 7: 配置值存儲為明文，缺少完整性校驗
- **嚴重級別**: P4（低危）
- **涉及文件**: `vault/db.py` config 表相關
- **影響功能**: 系統配置
- **問題描述**:
  - 配置項（如 `embedding_provider`, `embedding_model`）以明文存儲在 SQLite 中
  - 缺少完整性校驗機制
  - 如果數據庫文件被篡改，可能導致配置被惡意修改
  - 例如將嵌入模型改為惡意模型路徑
- **修復建議**:
  1. 對關鍵配置項添加完整性校驗（如哈希驗證）
  - 配置變更時記錄審計日誌
  2. 敏感配置（如 API Key）使用操作系統鑰匙圈存儲
- **當前風險評估**: 低。本地文件架構下，數據庫文件通常已受操作系統保護。

#### 問題 8: YAML 解析使用 `safe_load` 但仍有邊際風險
- **嚴重級別**: P4（低危）
- **涉及文件**: `vault/compiler.py`, `vault/graph.py`
- **影響功能**: 實體規則加載、Markdown frontmatter 解析
- **問題描述**:
  - 代碼中已使用 `yaml.safe_load()` 而非 `yaml.load()`，這是正確的
  - 但 `safe_load` 仍然可以構造一些 Python 基本對象
  - 對於深度嵌套或極大的 YAML 文件，可能導致內存消耗或解析超時
  - 存在 Billion Laughs 攻擊的潛在風險（儘管 PyYAML 有一定防護）
- **修復建議**:
  1. 限制 YAML 解析的最大深度
  2. 限制輸入文件大小
  3. 考慮使用更嚴格的 YAML 解析器
- **當前風險評估**: 低。用戶通常導入自己的文件，攻擊動機不足。

## v3.4 問題修復確認

### ✅ 已修復: P2 - XSS 注入風險
- **原問題**: `_generate_snippet` 高亮功能存在 XSS 注入
- **修復狀態**: 已完全修復
- **修復措施**:
  1. 添加 `escape_html` 參數，默認為 `True`
  2. 高亮前對文本進行 HTML 實體轉義
  3. 查詢詞也進行 HTML 轉義
- **驗證結果**: 安全測試通過，惡意腳本標籤被正確轉義

### ✅ 已修復: P2 - `highlight_tag` 標籤注入
- **原問題**: `highlight_tag` 參數無驗證，可注入任意 HTML
- **修復狀態**: 已完全修復
- **修復措施**:
  1. 引入 `ALLOWED_TAGS` 白名單（`em`, `strong`, `mark`, `span` 等 10 種安全標籤）
  2. 無效標籤名自動降级為 `em`
  3. 標籤名強制轉小寫
- **驗證結果**: 安全測試通過，惡意標籤名被正確攔截

### ✅ 已修復: P3 - 深分頁 DoS 風險
- **原問題**: `offset` 可達 9999，結合 `limit` 可能處理大量數據
- **修復狀態**: 已修復
- **修復措施**:
  1. `MAX_OFFSET` 從 9999 降至 2000
  2. 添加 `MAX_SEARCH_WINDOW = MAX_OFFSET + MAX_LIMIT` 限制
  3. 確保總查詢量不會過大
- **驗證結果**: 分頁查詢性能穩定，最大查詢窗口可控

### ✅ 已修復: P3 - 快取鍵值衝突風險
- **原問題**: 使用 `|` 分隔可能導致緩存鍵衝突
- **修復狀態**: 已完全修復
- **修復措施**:
  1. 改用 JSON 序列化 + MD5 哈希生成緩存鍵
  2. `sort_keys=True` 確保參數順序不影響鍵值
  3. 徹底杜絕鍵值衝突可能
- **驗證結果**: 相同參數生成相同鍵，不同參數不會衝突

### ✅ 已修復: P3 - 快取內存使用無限制
- **原問題**: 只有條目數限制，沒有內存大小限制
- **修復狀態**: 已修復
- **修復措施**:
  1. 添加 `_max_cache_memory_mb` 配置（預設 32MB）
  2. 實現 `_estimate_result_size` 方法估算內存占用
  3. 實現 `_evict_to_memory_limit` 方法進行內存驅逐
  4. 單條結果超過總限制時直接跳過緩存
- **驗證結果**: 內存使用量可控，不會出現內存溢位

### ✅ 已修復: P2 - 圖譜擴展權限繞過
- **原問題**: 圖譜擴展可能返回超出權限範圍的內容
- **修復狀態**: 已完全修復
- **修復措施**:
  1. `get_neighbors` 方法內置 SQL 層級的權限過濾
  2. `_apply_graph_expand` 進行了第二重權限驗證
  3. 支持 `min_trust`、`layer`、`category` 三種維度的權限控制
- **驗證結果**: 12 項安全測試全部通過，圖譜擴展嚴守權限邊界
