### P1 - 嚴重問題

*本版本未發現 P1 級別嚴重安全問題。*

### P2 - 高危問題

#### 問題 1: `_generate_snippet` 關鍵詞高亮存在 XSS 注入風險
- **嚴重級別**: P2（高危）
- **涉及文件**: `vault/search.py` 第 2075-2082 行
- **影響功能**: 片段生成 (`include_snippet`) + 高亮 (`highlight_snippet`)
- **問題描述**:
  - 當 `highlight=True` 時，查詢詞被直接用於構造 HTML 標籤
  - 代碼：`f'<{highlight_tag}>\\g<0></{highlight_tag}>'`
  - 雖然查詢詞使用 `re.escape` 進行了正則轉義，但**沒有進行 HTML 實體轉義**
  - 如果查詢詞包含諸如 `<script>`、`"onmouseover="` 等內容，且文本中確實存在這些詞，則生成的 HTML 片段可能導致 XSS
  - 攻擊場景：攻擊者先將惡意內容寫入知識庫，再通過包含相同惡意詞的查詢觸發高亮，從而在顯示結果的頁面執行惡意代碼
- **驗證代碼**:
  ```python
  # 假設知識庫中有內容: "測試 <img src=x onerror=alert(1)> 內容"
  # 查詢: "<img src=x onerror=alert(1)>"
  # 高亮後輸出: "... <em><img src=x onerror=alert(1)></em> ..."
  # 這會導致圖片標籤被解析並執行 onerror
  ```
- **修復建議**:
  1. 在高亮替換前，對文本中的 HTML 特殊字符進行實體轉義（`&`, `<`, `>`, `"`, `'`）
  2. 或者先高亮再轉義（注意順序：先高亮後轉義會破壞高亮標籤）
  3. 正確順序：先對原始文本進行 HTML 轉義，再進行高亮替換

#### 問題 2: `highlight_tag` 參數無驗證，存在標籤注入風險
- **嚴重級別**: P2（高危）
- **涉及文件**: `vault/search.py` 第 1996-2000 行
- **影響功能**: 片段高亮功能
- **問題描述**:
  - `highlight_tag` 參數直接用於構造 HTML 標籤，沒有任何驗證
  - 代碼：`f'<{highlight_tag}>\\g<0></{highlight_tag}>'`
  - 如果外部調用者可以控制此參數，可能注入任意 HTML 屬性或標籤
  - 例如：`highlight_tag='img src=x onerror=alert(1)><span'` 會導致
    ```html
    <img src=x onerror=alert(1)><span>匹配詞</img src=x onerror=alert(1)><span>
    ```
- **修復建議**:
  1. 添加白名單驗證，只允許安全的標籤名（如 `em`, `strong`, `mark`, `span`, `b`, `i`）
  2. 或使用正則驗證標籤名格式：`/^[a-zA-Z][a-zA-Z0-9_-]*$/`
  3. 預設使用 `em` 標籤，不接受外部自定義

### P3 - 中危問題

#### 問題 3: 分頁功能 offset + limit 可能導致大量數據處理
- **嚴重級別**: P3（中危）
- **涉及文件**: `vault/search.py` 第 1499-1501 行
- **影響功能**: 分頁搜索
- **問題描述**:
  - 為了實現分頁，代碼使用 `limit = min(limit + offset, MAX_LIMIT + MAX_OFFSET)`
  - 當 `limit=500, offset=9999` 時，實際查詢量為 10499 條
  - 遠超 `MAX_LIMIT=500` 的單次查詢限制
  - 雖然最終只返回 `limit` 條結果，但中間過程需要處理大量數據
  - 結合圖譜擴展時，性能影響更顯著
- **影響範圍**: 性能/DoS 風險，需要惡意用戶頻繁調用大 offset 查詢
- **修復建議**:
  1. 降低 `MAX_OFFSET` 到合理範圍（如 1000）
  2. 或者對 `limit + offset` 設置更低的上限
  3. 對於深分頁，建議使用游標分頁（cursor-based pagination）而非 offset

#### 問題 4: 快取鍵值構造存在衝突風險
- **嚴重級別**: P3（中危）
- **涉及文件**: `vault/search.py` 第 1224-1229 行
- **影響功能**: 查詢快取
- **問題描述**:
  - `_get_cache_key` 使用 `|` 字符連接查詢和參數
  - 如果參數值中包含 `|` 字符，可能導致緩存鍵衝突
  - 例如：`query="a|b"` 和 `query="a"` + 某參數=`b` 可能生成相同的鍵
  - 影響：返回錯誤的緩存結果，導致信息泄露或業務邏輯錯誤
- **修復建議**:
  1. 使用更安全的分隔符（不太可能出現在查詢中的字符組合）
  2. 或者對參數值進行轉義/編碼（如 URL 編碼）
  3. 或者使用 JSON 序列化 + 哈希的方式生成緩存鍵

#### 問題 5: 快取缺少內存使用量限制
- **嚴重級別**: P3（中危）
- **涉及文件**: `vault/search.py` 快取相關代碼
- **影響功能**: 查詢快取
- **問題描述**:
  - 雖然有 `_cache_size` 限制條目數量，但單條緩存條目的大小沒有限制
  - 如果查詢返回大量大體積結果（如長文檔），可能導致內存占用過高
  - 每個結果包含完整的 `content_raw` 字段，可能很大
- **修復建議**:
  1. 添加單條緩存大小限制
  2. 或者添加總內存使用量限制
  3. 對於大結果，可以考慮只緩存 ID 和分數，不緩存完整內容

#### 問題 6: 搜索結果默認返回完整 `content_raw`，存在信息泄露風險
- **嚴重級別**: P3（中危）
- **涉及文件**: 所有搜索方法
- **影響功能**: 搜索結果返回
- **問題描述**:
  - 搜索結果默認返回完整的文檔內容（`content_raw` 字段）
  - 對於大型知識庫，單條結果可能包含數千字節的敏感內容
  - 如果接口被未經授權訪問，可能導致大量信息泄露
  - 雖然這是設計使然，但從安全最小化原則角度看有優化空間
- **修復建議**:
  1. 提供 `compact`/`minimal` 模式，只返回 ID、標題、摘要等必要字段
  2. 文檔內容按需加載（通過單獨的獲取詳情接口）
  3. 對於公開場景，預設使用緊湊模式

#### 問題 7: `_generate_snippet` 存在潛在 ReDoS 風險
- **嚴重級別**: P3（中危）
- **涉及文件**: `vault/search.py` 第 2076-2080 行
- **影響功能**: 片段生成與高亮
- **問題描述**:
  - 在循環中多次調用 `re.compile(re.escape(term), re.IGNORECASE)`
  - 雖然 `re.escape` 防止了正則注入，但大量長詞仍可能導致性能問題
  - 當查詢包含很多長詞時（最多 5 個），每個詞都要在整個片段中進行替換
  - 結合多次 `find` 操作，最壞情況下時間複雜度為 O(n * m)（n=文本長度, m=詞數）
- **影響範圍**: 性能影響較小，因為有查詢長度限制（1000 字符）和詞數限制（5 個）
- **修復建議**:
  1. 預先編譯所有模式，或使用單一正則表達式匹配所有詞
  2. 使用 `re.sub` 一次替換多個匹配
  3. 當前限制條件下風險可控，優先級可適當降低

## 新功能安全審查

### 分頁功能 (offset 參數)

**安全評分**: 4.0 / 5.0

✅ **優點**:
- offset 有邊界驗證（0 - 9999）
- 負數 offset 會被重置為 0
- 非整數 offset 會被重置為 0
- 有最大偏移量限制（MAX_OFFSET = 9999）

⚠️ **風險點**:
- `limit + offset` 可能遠超 MAX_LIMIT，導致中間處理量大
- 圖譜擴展後的分頁可能不準確（擴展改變了結果順序）
- 深分頁性能問題（見 P3-3）

**建議**: 降低 MAX_OFFSET 或使用游標分頁

### 快取功能

**安全評分**: 4.2 / 5.0

✅ **優點**:
- TTL 過期機制完善
- LRU 風格的舊條目驅逐
- 深拷貝防止外部修改影響緩存
- 可配置開關、大小、過期時間
- 快取命中統計

⚠️ **風險點**:
- 緩存鍵構造可能有衝突（見 P3-4）
- 缺少內存使用量限制（見 P3-5）
- 沒有緩存失效機制（數據更新時不清空相關緩存）

**建議**: 優化緩存鍵生成，添加內存限制

### 分數標準化功能

**安全評分**: 4.8 / 5.0

✅ **優點**:
- 正確的 min-max 標準化算法
- 保存原始分數到 `_original_score`
- 正確處理所有分數相同的邊界情況
- 可選開關，預設關閉保持向後兼容
- 分數被限制在合理範圍內

⚠️ **注意事項**:
- 標準化是基於當前查詢結果的分佈，不同查詢的分數不具可比性
- 這是預期行為，但需要在文檔中說明

**總體**: 實現正確，安全風險很低

### 片段生成與高亮功能

**安全評分**: 3.5 / 5.0

✅ **優點**:
- 多詞匹配加權算法合理
- 上下文窗口提取正確
- 省略號標記處理得當
- 查詢詞數量限制（最多 5 個）

⚠️ **嚴重風險**:
- XSS 注入漏洞（見 P2-1）
- `highlight_tag` 參數無驗證（見 P2-2）
- 沒有 HTML 實體轉義

**建議**: 緊急修復 XSS 問題，添加 HTML 轉義和標籤白名單

### 圖譜遍歷權限控制

**安全評分**: 4.5 / 5.0

✅ **優點**:
- SQL 層面進行權限過濾，減少側信道風險
- 雙重檢查：SQL 層 + Python 層（過濾擴展結果時再次驗證）
- 深度限制（MAX_DEPTH = 10）
- 鄰居數量限制（MAX_NEIGHBORS = 200）
- 支持 min_trust、layer、category 多維度權限控制
- 批量查詢優化，減少數據庫交互次數

⚠️ **輕微風險**:
- 仍然存在細微的側信道時序差異（完全消除需要常時間響應，代價太高）
- Python 層的二次檢查有輕微性能開銷

**總體**: 實現優秀，在安全和性能之間取得了良好平衡
