# Vault-for-LLM v0.6.24 Short Announcement Draft

Vault-for-LLM is now available on PyPI, and the next agent-install focused
release line makes it less like a single-agent side tool and more like a shared
project memory layer that different agents can use together.

Before posting this externally, set the version below to the exact PyPI build
that contains `vault setup-agent`.

What changed:

- Hermes Agent, OpenClaw, Codex, OpenCode, Claude Code, n8n, and generic
  MCP-compatible agents can all connect through the same local CLI/MCP pattern.
- Agents can now ask whether a Vault should be shared or private during setup.
- Existing Obsidian notes can be imported back into Vault with a dry-run first,
  then compiled into the local SQLite memory database after confirmation.
- If Obsidian is part of the workflow, agents can generate cron, LaunchAgent,
  or n8n sync templates so new notes keep flowing into Vault.
- The package is available from PyPI as `vault-for-llm`.

The point is not to replace Obsidian, Hermes memory, Mem0, Zep, or a wiki.
Vault-for-LLM is the project-memory governance layer between human notes,
repo docs, and AI agents: search first, read bounded source ranges, propose
before storing, cite the source, and keep the memory recoverable.

Install:

```bash
pip install "vault-for-llm[mcp]"
vault setup-agent
```

For agent-driven setup:

```text
Install Vault-for-LLM for this project. Ask whether the database should be
shared or private, ask which optional features to enable, ask whether I have an
Obsidian vault to import, configure CLI/MCP, and run a search/read/propose smoke
test.
```
