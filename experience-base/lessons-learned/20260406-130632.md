## Lesson #33: 記憶系統自動蒸餾機制

[Situation] MEMORY.md 膨脹到 1021 行（51KB），影響系統效率和 Token 消耗。

[Wrong] 沒有自動蒸餾機制，依賴手動整理；記憶只增不減，無法保持高效狀態。

[Correct] 實裝 autoDream 記憶蒸餾系統：
- 配置：`config/autoDream.yaml`, `scripts/autoDream.sh`
- 觸發條件：每天 03:00 或超過 500 行
- 保留策略：score≥9 原樣保留，score≥7 保留摘要，scar 格式特殊保留
- 成果：1021 行 → 82 行（93.6% 壓縮）

[Behavior Change] 記憶系統需要自動化維護，不能只增不減；設置自動蒸餾機制是必要的基礎設施。

[Activation] 當記憶文件超過 500 行，或創建新的記憶系統時。

## Lesson #34: Token 節省優化策略

[Situation] Heartbeat 每隔 15/30 分鐘執行完整檢查，消耗大量 Token（估計每天 $20）。

[Wrong] 所有檢查都使用 Full Mode，頻率過高，不分場景使用相同的 Token 模式。

[Correct] 實施時間分段和 Token 模式分類：
- Task Status Check: 15分 → 60分（節省 75%）
- Supabase Tasks: 30分 → 60分（節省 50%）
- 時間分段：深度睡眠（23:00-07:00）、晨間過渡、全速執行、節能模式
- Token 模式：Full vs Light，根據任務重要性選擇
- 成果：預計總節省 60-70%

[Behavior Change] 經常性任務需要優化頻率和模式，節省成本是重要的系統設計考量。

[Activation] 設計定期檢查任務、心跳機制，或發現 Token 消耗過高時。

## Lesson #35: 工具腳本需要調度配置

[Situation] Guardrails 百科全書有同步腳本（`auto-sync.sh`），但缺少定時任務配置；最後提交是 11 天前。

[Wrong] 創建工具腳本後未配置到調度系統（HEARTBEAT/cron），導致腳本無法自動執行。

[Correct] 將 Guardrails 同步添加到 HEARTBEAT.md，配置為每 60 分鐘自動執行（Light Mode）。

[Behavior Change] 工具腳本創建後必須配置到調度系統中才能發揮作用；檢查腳本是否存在是不足夠的。

[Activation] 創建或發現任何工具腳本、自動化腳本時，立即檢查調度配置。

## Lesson #36: 系統健康檢查標準化

[Situation] 需要全面了解系統狀態，但檢查零散，缺乏整體視圖。

[Wrong] 零散檢查各個組件，沒有標準化的健康檢查流程和報告格式。

[Correct] 建立標準化健康檢查流程：
- Gateway 狀態
- Agent 會話狀態（主會話 + 子代理）
- 活躍專案 PM 狀態
- 系統組件（腳本、配置）
- 智能優化狀態
- 生成結構化報告

[Behavior Change] 系統健康檢查應該標準化、結構化，能快速定位問題並評估整體健康度。

[Activation] 當需要評估系統狀態，或定期維護時。
