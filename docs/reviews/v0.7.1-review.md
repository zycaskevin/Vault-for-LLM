# Vault-for-LLM v0.7.1 Review Notes

Date: 2026-06-25
Version reviewed: 0.7.1

## Summary

v0.7.1 is a stability patch for the first stable v0.7 line. Its direct change is
install and smoke-test hardening after the v0.7.0 PyPI closeout found two CLI
details that agents could easily get wrong:

- active notes use `vault add "Title" --content "..."`
- bounded source reads use `vault map read <knowledge_id> --lines START-END`

The broader v0.7 line is the Agent Knowledge Platform foundation: candidate-first
memory, Document Map bounded reads, Search QA, session capture, automation
cycles, multi-agent setup, and MCP startup/read/propose tools.

## Corrected Interpretation

`vault add` does not replace `vault remember`.

- `vault add` writes directly to active knowledge. Use it for trusted manual
  notes, verified imports, and install smoke tests.
- `vault remember` creates a memory candidate. Use it for agent-generated memory
  and automation flows that should pass privacy, duplicate, metadata, and quality
  gates before promotion.

`vault map read <knowledge_id> --lines START-END` requires a real knowledge ID.
The expected flow is:

```bash
vault search "keyword" --json
vault map show <result_id>
vault map read <result_id> --lines 1-20
```

Guessing an arbitrary ID can correctly return "Knowledge id not found".

## Positive Findings

- Agent-driven install is now easier to validate because generated
  `local-smoke.sh` checks active add/search, Document Map bounded read,
  candidate-first remember, candidate listing, and core MCP startup tools.
- README command smoke now covers `map build/read`, so the documented bounded
  read path is protected in CI.
- The v0.7 line keeps memory writes conservative: automation remains
  report-first by default, and agent-generated memory stays candidate-first
  unless explicitly reviewed or narrowly policy-promoted.
- Search QA and Document Map provide measurable retrieval checks instead of
  relying only on subjective "the agent remembered it" claims.

## Follow-Up Risk: sqlite-vec Runtime Availability

Some Linux or container SQLite builds can have the `sqlite-vec` Python package
installed while loadable extension execution is blocked by the SQLite runtime,
for example with `sqlite3.OperationalError: not authorized`.

This should be treated as a runtime capability issue, not as data loss.

- Keyword and FTS search should continue to work.
- JSON-backed `semantic_vectors` can still exist and be counted.
- Legacy sqlite-vec shadow indexes may be unavailable in that environment.
- `vault doctor` and `vault stats` should clearly report this degraded state so
  users do not mistake it for missing embeddings.

## Public Messaging

Use this wording:

> v0.7.1 is the install and CLI smoke hardening patch for the stable v0.7 line.
> v0.7 as a line turns Vault-for-LLM into an Agent Knowledge Platform foundation:
> candidate-first memory, bounded reads, Search QA, session capture, automation
> cycles, and multi-agent startup tooling.

Avoid this wording:

> `vault add` replaces `vault remember`.

Avoid claiming that every v0.7 platform feature was added specifically in
v0.7.1. v0.7.1 is a patch release on top of the v0.7 platform line.
