# Decision Record: Enter v0.7 Release Candidate

Date: 2026-06-25
Version: v0.7.0rc1

## Context

The 0.6 line added and hardened the pieces needed for Vault-for-LLM to act as a local-first agent knowledge platform:

- agent-driven installation
- MCP startup, search, bounded read, memory proposal, and automation tools
- multi-agent update rollout health
- candidate-first memory governance
- automation briefs, inboxes, handoff, and reversible lifecycle actions
- Supabase remote-read support as an optional bridge, while SQLite remains the source of truth
- module-size gates and repeated MCP module splits before the 0.7 boundary

Continuing to ship only 0.6 patch releases would hide the product-level transition from tool collection to platform foundation.

## Decision

Start the v0.7 release-candidate line with `0.7.0rc1`.

This RC is not a new feature expansion. It is a packaging, install, documentation, and platform-boundary validation release before stable `0.7.0`.

## Release Tag Shape

Use PyPI-compatible PEP 440 versions:

- package version: `0.7.0rc1`
- GitHub tag: `v0.7.0rc1`

The release parity gate now accepts release-candidate tags in addition to stable `vX.Y.Z` tags.

## Safety Notes

This RC does not add a new memory mutation path, hosted dependency, MCP permission, auto-promote path, remote sync behavior, or background automation trigger.

The stable v0.7.0 release should ship only after clean package install, Agent setup smoke, MCP tool smoke, privacy scan, module-size gate, README command smoke, and release parity all pass on the RC line.

## Consequences

Future 0.7 work should focus on product boundary, safety review, installer clarity, and platform documentation before adding broad new capability.
