# 結構化討論框架（PREP）

## 用途
用於系統設計、技術選型、產品決策的結構化討論

## 框架結構

### Point（觀點）
直接給出你的建議或結論

### Reason（原因）
為什麼這樣建議？分析優缺點

### Example（例子）
給出具體案例或場景

### Point（重申觀點）
總結強調觀點

## 使用示例

### 問題：應該用什麼技術框架？

**Point**: 建議使用持久化會話模式

**Reason**:
- 工作區隔離性：每個智能體有獨立的 workspace
- 記憶持久化：智能體「睡覺」後喚醒，記憶還在
- 協作效率：A2A 協議開銷小
- 易於管理：可以檢查每個智能體的狀態

**Example**:
```
結構：
/workspace/agents/
├── cto/（CTO 的工作區）
├── developer/（開發員的工作區）
└── eve/（我的工作區）

協作：
sessions_send(sessionKey: "cto", message: "請審查這個架構設計...")
```

**Point**: 持久化會話模式是最優選擇，兼顧隔離性、持久性、協作效率和管理便捷性。

## 注意事項

1. 避免憑直覺決策，每個建議都要有原因
2. 例子要具體，最好給出代碼或配置示例
3. 如果有多個選項，可以比較每個選項的優缺點
4. 重申觀點時要簡潔明確

## 適用場景

- 系統架構選型
- 技術框架選擇
- 設計模式選擇
- 流程機制設計
