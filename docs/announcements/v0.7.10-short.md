# Vault-for-LLM v0.7.10

This patch completes the startup path for the automatic memory-ingestion
receipt.

`vault automation handoff` and MCP `vault_automation_handoff` now attach
`pipeline_receipt_content` when `reports/automation/pipeline-latest.md` or
`.json` exists. The startup read order is now:

1. `fleet_health_content`
2. `pipeline_receipt_content`
3. `review_summary_content`
4. `learning_health_content`
5. `content`

The selected handoff `content` stays unchanged for existing cycle/inbox readers.
The pipeline receipt remains read-only and is only a compact ingestion receipt;
it does not expose raw transcript content, candidate body fields, or gate
payloads.

Internally, handoff assembly moved into `vault/automation_handoff.py` so the
main automation workflow module stays below the module-size guardrail.
