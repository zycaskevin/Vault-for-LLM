# Vault-for-LLM v0.6.87

This release makes multi-runtime installs easier to operate after setup.

`vault setup-agent` now writes:

- `agent-install/README-runtime-update-playbook.md`
- `agent-install/runtime-update-playbook.json`

The playbook answers the common local-machine question:

> If Codex, Claude Code, OpenClaw, Hermes Agent, and other runtimes all use the
> same Vault project, how do they know who upgraded, who should read the shared
> notice, and who should restart or upgrade?

The rule is intentionally conservative:

1. each runtime reads its own focused status;
2. missing status falls back to a no-network local status check;
3. unclear rollout state uses MCP `vault_update_status` with `doctor=true`;
4. compact automation handoff is read before deeper memory search;
5. no runtime auto-upgrades or restarts another runtime.

This keeps Vault-for-LLM as the shared memory layer while letting different
Agent runtimes remain replaceable hands around the same project brain.
