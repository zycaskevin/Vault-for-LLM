# Decision Record: v0.7 Stabilization Before the Next Release

Date: 2026-06-26

## Context

Vault-for-LLM has reached a broad v0.7 feature surface: MCP profiles, memory
automation, session capture, temporal metadata, Supabase sync, Obsidian import,
and multi-Agent startup artifacts are all present.

The next risk is not missing features. The next risk is that the project looks
noisy from the outside because every maintenance improvement becomes a release.
Mature projects often merge many focused PRs before publishing one meaningful
version. Vault should follow that pattern for the next stabilization window.

## Decision

Do not publish the next external package release until the v0.7 stabilization
queue is complete.

The queue is:

1. Continue splitting large modules around stable review boundaries.
2. Reduce README noise and move detailed usage into focused docs.
3. Make MCP tool-profile guidance clearer, with `core` as the default agent
   recommendation.
4. Tighten the automation review surfaces so the human-facing 5% queue stays
   short, deduped, and easy to approve or reject.
5. Integrate temporal memory behavior more deeply into search and user-facing
   guidance.
6. Strengthen Supabase and multi-Agent remote-reader safety validation.
7. Add a repeatable install-smoke matrix that checks source checkout, clean
   wheel install, MCP stdio, automation, setup-agent, migration, Obsidian, and
   optional remote-reader paths.

These tasks should land as focused PRs on `main`. They do not require a version
bump by themselves.

## Release Gate

Before the next release, the maintainer should be able to point to evidence for
each item in the queue:

- merged PR links or commits,
- local full-test results,
- CI results,
- clean wheel-install smoke output,
- public/privacy gate output,
- README command smoke output,
- and any remote-reader or Supabase safety smoke that applies.

If the release contains only internal refactors and documentation cleanup, it
may be batched into a later stability release instead of published immediately.

## Safety Notes

This decision does not add a new memory mutation path, background automation
trigger, remote sync permission, MCP tool, or hosted dependency. It is a release
cadence and stabilization rule.

Automation remains candidate-first and report-first unless the user explicitly
enables a stronger policy. Remote/shared memory remains governed by scope,
sensitivity, owner, and allowed-agent metadata.

## Deferred

This decision does not choose a future version number. The next release can be a
patch or minor stability release depending on the final user-visible scope.
