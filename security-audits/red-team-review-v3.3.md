## 新发现与遗留问题

### P2 — 中危问题

#### P2-1: `search_vector` 两步查询的侧信道风险

**位置**: `vault/db.py:877` — `search_vector()` 方法

**原理**:
- 向量搜索采用两步查询：
  1. 先从 `knowledge_vec` 表按距离取 top-K（无权限过滤）
  2. 再从 `knowledge` 表按权限过滤
- 存在时序侧信道风险：攻击者可通过响应时间推断受限文档的存在性和大致数量
- 如果 top-K 结果中大部分是低信任级文档，最终返回结果可能远少于 limit，也会泄露信息

**攻击场景**:
- 攻击者通过构造特定查询，观察返回结果数量与预期 limit 的差异，推断受限文档的存在
- 例如：查询敏感主题时，如果返回结果远少于 limit，可能暗示存在大量受限文档

**风险等级**: P2 — 侧信道信息泄露，实际利用难度较高

**修复建议**:
```python
# 方案：使用子查询或 JOIN 在单条 SQL 中完成权限过滤 + 向量匹配
# 由于 sqlite-vec 的限制，可能需要采用其他策略：
# 1. 增大第一步的 limit（如 limit * 5），降低侧信道准确性
# 2. 或者使用随机化响应时间
# 3. 或者将信任等级预存入向量表（如果信任等级不频繁变化）
```

---

#### P2-2: `get_neighbors` 图遍历权限后过滤可能泄露图结构

**位置**: `vault/db.py:1055` — `get_neighbors()` 方法

**原理**:
- BFS 遍历时未考虑权限，先找出所有可达邻居，再进行权限过滤
- 这意味着：
  1. 遍历过程中可能经过低信任/受限节点到达高信任节点
  2. 如果低信任节点是关键桥接点，过滤后可能丢失合法的高信任邻居
  3. 时序侧信道：遍历时间反映了总邻居数（含受限节点）
- 与 P2-1 类似，但图结构信息的敏感性更高

**攻击场景**:
- 攻击者通过图扩展的响应时间和结果数量差异，推断受限节点的存在和连接情况

**风险等级**: P2 — 图结构信息泄露，实际利用需要结合上下文

**修复建议**:
```python
# 在 BFS 遍历时就进行权限过滤，只扩展有权限访问的节点
# 注意：这需要在每个节点扩展前查询其权限属性
# 或者：将 trust/layer/category 冗余存储到 edges 表或节点表
```

#### P2-3: LLM 查询改写的注入防护可被绕过

**位置**: `vault/search.py` — `_rewrite_query_with_llm()` 方法

**原理**:
- 当前注入检测使用简单的字符串匹配（`ignore previous`、`忘記之前` 等）
- 攻击者可通过以下方式绕过：
  - 使用同义词或变体表达（如「忽略之前的」、「請無視」）
  - 使用编码或混淆（如 Unicode 变体、base64）
  - 使用多轮嵌套指令
  - 使用不同语言混合

**现有防护层次**:
1. ✅ 输入长度限制（500 字符）
2. ⚠️ 注入模式检测（字符串匹配，可绕过）
3. ✅ 系统提示强化 + XML 边界隔离
4. ✅ 输出验证（长度限制、内容检查）

**风险等级**: P2 — 虽然有多层防护，但第一层检测可被绕过，增加了成功注入的概率

**修复建议**:
```python
# 1. 扩展注入模式检测的覆盖范围（更多语言、更多变体）
# 2. 添加规范化步骤（统一大小写、去除特殊字符、简繁转换）后再检测
# 3. 考虑使用更智能的检测方式（如嵌入相似度匹配）
# 4. 强化系统提示中的安全指令（sandwich prompt 技术）
```

#### P3-2: `cross_encoder_model` 参数未验证格式

**位置**: `vault/search.py` — `VaultSearch.__init__()` 及 `CrossEncoderReranker`

**原理**:
- `cross_encoder_model` 参数可以是任意字符串
- 虽然目前仅用于模型加载（通过 sentence-transformers 或 onnxruntime），不会造成直接安全问题
- 但如果未来支持从任意路径加载模型，可能引入路径遍历或任意文件加载风险
- 当前环境下风险极低，因为模型加载机制本身有安全限制

**风险等级**: P3 — 防御性编程建议，当前无实际风险

**修复建议**:
```python
# 添加基础格式验证，防止意外值
import re
if not re.match(r'^[a-zA-Z0-9_\-/]+$', cross_encoder_model):
    raise ValueError(f"Invalid cross_encoder_model name: {cross_encoder_model}")
```

#### P3-3: 向量搜索维度未验证

**位置**: `vault/db.py:877` — `search_vector()` 方法

**原理**:
- 传入的 `query_embedding` 长度未与数据库中存储的向量维度进行比对

```python
# 在搜索前验证维度
expected_dim = int(self._get_config("embedding_dim", "384"))
if len(query_embedding) != expected_dim:
    raise ValueError(f"Embedding dimension mismatch: expected {expected_dim}, got {len(query_embedding)}")
```

#### P3-4: LIKE 查询通配符注入（性能影响）

**位置**: `vault/search.py` — `_search_keyword_like()` 方法

**原理**:
- 用户输入直接用于 LIKE 查询：`pattern = f"%{term}%"`
- 如果 term 包含 `%` 或 `_` 等 LIKE 通配符，会改变查询语义
- 恶意构造的通配符模式可能导致全表扫描，性能下降
- 但由于有 LIMIT 限制且仅用于 fallback 路径，实际影响有限

**风险等级**: P3 — 低风险性能影响，仅在 FTS5 不可用时触发

**修复建议**:
```python
# 转义 LIKE 特殊字符
def _escape_like_pattern(term: str) -> str:
    return term.replace('%', '\\%').replace('_', '\\_')
# 使用时指定 escape 字符
"title LIKE ? ESCAPE '\\'"
```

## 总结

Vault-for-LLM 搜索功能的安全性在 v3.2 基础上有显著提升，核心的注入防护、资源限制、权限边界等方面已达到较高水平。v3.2 发现的 2 个 P1 高危问题全部修复，4 个 P2 中危问题大部分修复，整体评分从 4.1/5 提升至 4.4/5。

当前剩余的主要风险集中在：
1. **侧信道信息泄露** — 向量搜索和图遍历的两步查询模式可能通过响应时间泄露受限数据的存在性
2. **LLM 注入防护深度** — 第一层模式检测可被绕过，需依赖后续防御层

这些问题均为中低风险，在大多数使用场景下不会造成实际危害，但对于安全性要求较高的部署场景，建议优先修复 P2 级别的侧信道问题。

**整体安全等级：良好** — 核心安全机制完善，剩余问题均为边缘场景的渐进式优化。
